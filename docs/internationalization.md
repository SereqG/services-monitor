# Internationalization (English / Polish)

The app supports switching the UI between **English (default) and Polish**. The English copy is
the canonical source; Polish is authored as natural, user-friendly text (not a literal
translation).

There are three layers of user-facing text, handled differently:

## 1. Static UI text — frontend dictionary

- Location: `frontend/lib/i18n/`.
  - `en.ts` — the canonical dictionary and its `Dictionary` type. Parameterized strings are
    functions (e.g. `foundPages(count, host)`) for type-safe interpolation.
  - `pl.ts` — `pl: Dictionary`. TypeScript enforces that Polish has **every** key English has, so
    there are no missing-key gaps to fall back on.
  - `context.tsx` — `LanguageProvider` + `useI18n()` hook, exposing `{ lang, setLanguage, dict }`.
- Persistence: the choice is stored in a `lang` cookie (one year). The **root layout**
  (`app/layout.tsx`, a Server Component) reads the cookie with `await cookies()` and passes the
  initial language to the provider, and sets `<html lang>` — avoiding a hydration flash. Reading
  the cookie opts the route into dynamic rendering (expected). Setting the cookie happens
  client-side in `setLanguage` (cookies cannot be set during Server Component render).
- Switching: the **Header** (`components/layout/Header.tsx`) hosts the EN/PL toggle. This is the
  single global control; the AI-summary language follows it.
- Components read text via `const { dict } = useI18n()` and reference `dict.<section>.<key>`.

## 2. Backend deterministic findings — localized in the frontend by `code`

Analyzer findings (SEO / security / accessibility) are generated by the backend in **English**,
which stays the deterministic source of truth. To make them localizable without changing that:

- Every finding carries a stable `code`, and parameterized findings also carry structured
  `params` (e.g. `{"header": "Content-Security-Policy"}`, `{"length": 12, "min": 30}`). The
  English `message`/`title` is left exactly as-is.
- The frontend maps `code` → a Polish template in `frontend/lib/i18n/findings.ts` and interpolates
  `params`. In English mode it renders the backend text directly; for any code without a Polish
  template it **falls back** to the backend English (explicit, never blank).

Relevant backend schemas: `SecurityFinding` (`code`, `params`), `SeoIssue` (`params`),
`AccessibilityIssue` (already has `code` + `count`).

## 3. AI summary — generated in the chosen language

See `docs/ai_audit_summary_architecture_claude_prompt.md` → *Output Language*. The audit request
carries `language`; the AI prose is generated in that language at audit time and stored on the
report. It is **not** regenerated when the UI language is toggled afterward.

## Known boundary

Streamed discovery/audit progress messages (SSE `event.message`) are ephemeral backend English
logs and are **not** localized.
